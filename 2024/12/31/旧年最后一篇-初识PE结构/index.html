前置

位域：在结构体中，位域用于存储多个 二进制标志（例如 0 或 1 的值）。每个标志都占用结构体中的一个或多个比特位，而不是像普通的整数那样占用完整的字节或字。在 C 和 C++ 中，位域允许通过指定每个字段所占的比特数，节省内存空间，特别是在需要存储多个二进制标志时。

名称 描述
地址 是 “虚拟地址” 而不是“物理地址”。为什么不是“物理地址”呢？因为数据在内存的位置经常在变，这样可以节省内存开支、避开错误的内存位置等的优势。同时用户并不需要知道具体的“真实地址”，因为系统自己会为程序准备好内存空间的（只要内存足够大）
镜像文件 包含以 EXE 文件为代表的 “可执行文件”、以DLL文件为代表的“动态链接库”。为什么用“镜像”？这是因为他们常常被直接“复制”到内存，有“镜像”的某种意思。
RVA 英文全称 Relatively Virtual Address。偏移（又称“相对虚拟地址”）。相对镜像基址的偏移。
节 节是PE文件中代码或数据的基本单元(即包含代码数据的容器)。原则上讲，节只分为 “代码节” 和 “数据节” ，但节的名称可以自定义
VA 英文全称 Virtual Address。虚拟地址

PE 文件的执行顺序

  1. 当一个 PE 文件 被执行时，PE 装载器 首先检查 DOS header 里的 PE header 的偏移量。如果找到，则直接跳转到 PE header 的位置。
  2. 当 PE装载器 跳转到 PE header 后，第二步要做的就是检查 PE header 是否有效。如果该 PE header 有效，就跳转到 PE header 的尾部。
  3. 紧跟 PE header 尾部的是节表。PE装载器执行完第二步后开始读取节表中的节段信息，并采用文件映射的方法将这些节段映射到内存，同时附上节表里指定节段的读写属性。

文件映射

在执行一个PE文件的时候，Windows并不在一开始就将整个文件读入内存，而是采用与内存映射的机制，也就是说，Windows装载器在装载的时候仅仅建立好虚拟地址和PE文件之间的映射关系，只有真正执行到某个内存页中的指令或者访问某一页中的数据时，这个页面才会被从磁盘提交到物理内存，这种机制使文件装入的速度和文件大小没有太大的关系

  1. PE文件映射入内存后，PE装载器将继续处理PE文件中类似 import table （输入表）的逻辑部分。

PE结构

1. 概述

PE(Portable Executable)，可移植可执行文件，类似于Linux系统中的ELF(Executable and Linkable Format)，是Windows系统中可执行文件的标准格式。

常见的PE文件格式有：.exe(executable)、.dll(dynamic link library)、.sys(system file)、.lib(library)。

其中主要有：

  • DOS头
  • NT头（也叫做PE头）
  • 节表

  1. DOS头（DOS Header）
    DOS头是 PE 文件的最开始部分，它是为了兼容 DOS 程序而设计的，现代的 Windows 程序不再使用 DOS 的代码，但 PE 文件为了兼容性保留了这一部分。

结构如下

DOS头分为

  • MZ头部（对应的文件最起始位置的magic number 4D5A，即e_magic字段，其解码的结果正是MZ，对应一个叫做微软元老级程序员mark zbicowski）
  • 一些如今也使用的字段，不过现在也不大发挥作用，如图中文件相关参数等
  • 一些保留字段，即图中的reserved字段和一些值为0的字段，在现在已经不发挥作用了，只是依旧保留在PE结构中
  • e_ifanew字段，类型为long，即图中最后的那个注释是NTheaderoffset,意味PE头相对于该文件起始位置的偏移量

你肯定看见了在图中dosheader的后面还有个叫做DOSstub的东西，这是一段有效的DOS程序，现在很少用，它在不支持PE文件格式的操作系统中（如MSDOS）会返回出错误提示

  1. NT头（NT Header，也叫做PE头）

它是IMAGE_NT_HEADER的简称，其整体是一个结构体，主要由三部分构成：

  1. signature字段，类型为long，其值为0x50450000，即PE00(图中可见)，当加载器检测解析PE文件时，会首先检查到这个特定的签名，然后会安照NT头相关的规范进行后续的解析

  2. FileHeader字段，类型为IMAGE_FILE_HEADER,是一个包含多种结构的结构体，定义如下

    typedef struct _IMAGE_FILE_HEADER {
   WORD    Machine;               // 目标机器架构
   WORD    NumberOfSections;      // 节的数量
   DWORD   TimeDateStamp;         // 文件的时间戳（编译时生成的时间）
   DWORD   PointerToSymbolTable;  // 指向符号表的偏移量（调试时用）
   DWORD   NumberOfSymbols;       // 符号数量（调试时用）
   WORD    SizeOfOptionalHeader;  // 可选头的大小
   WORD    Characteristics;       // 文件的特性
        } IMAGE_FILE_HEADER;

它包含了 PE 文件的元数据，主要用于帮助操作系统加载并执行文件。

Characteristics字段，它是一个位域，用1(true)和0(false)来表示该文件的特性，具体见上图

timedatestamp字段，它显示的时间是UTC时间

  1. OptionalHeader字段，类型为IMAGE_OPTIONAL_HEADER，它相当于是对fileheader的补充（虽然optional是可选的的意思，但实际上不能缺），其结构定义如下

         WORD    Magic;                     // 文件类型（PE32 或 PE32+,即PE64）
         BYTE    MajorLinkerVersion;         // 链接器的主版本号
         BYTE    MinorLinkerVersion;         // 链接器的次版本号
         DWORD   SizeOfCode;                 // 代码段的大小
         DWORD   SizeOfInitializedData;      // 初始化数据段的大小
         DWORD   SizeOfUninitializedData;    // 未初始化数据段的大小
         DWORD   AddressOfEntryPoint;        // 程序入口点地址（相对于加载基址）
         DWORD   BaseOfCode;                 // 代码段的基址（虚拟地址）
         DWORD   BaseOfData;                 // 数据段的基址（仅对于 PE32 格式有效）
         DWORD   ImageBase;                  // 加载的起始虚拟地址
         DWORD   SectionAlignment;           // 节对齐大小（内存中的对齐单位）
         DWORD   FileAlignment;              // 文件对齐大小（文件中的对齐单位）
         WORD    MajorOperatingSystemVersion; // 操作系统的主版本号
         WORD    MinorOperatingSystemVersion; // 操作系统的次版本号
         WORD    MajorImageVersion;          // 图像的主版本号
         WORD    MinorImageVersion;          // 图像的次版本号
         WORD    MajorSubsystemVersion;      // 子系统的主版本号
         WORD    MinorSubsystemVersion;      // 子系统的次版本号
         DWORD   Win32VersionValue;          // 保留字段，通常为 0
         DWORD   SizeOfImage;                // 图像的总大小（包括所有节和其他数据）
         DWORD   SizeOfHeaders;              // PE 文件头部（包括 DOS 头和 NT 头）大小
         DWORD   CheckSum;                   // 文件校验和（用于验证文件的完整性）
         WORD    Subsystem;                  // 程序的子系统类型（控制台应用程序、Windows GUI 等）
         WORD    DLLCharacteristics;         // DLL 特性标志（如支持动态加载）
         DWORD   SizeOfStackReserve;         // 保留堆栈大小
         DWORD   SizeOfStackCommit;          // 提交堆栈大小
         DWORD   SizeOfHeapReserve;          // 保留堆大小
         DWORD   SizeOfHeapCommit;           // 提交堆大小
         DWORD   LoaderFlags;                // 加载器标志（通常为 0）
         DWORD   NumberOfRvaAndSizes;        // 数据目录的数量（即，数据目录数组的大小）
         IMAGE_DATA_DIRECTORY DataDirectory[IMAGE_NUMBEROF_DIRECTORY_ENTRIES]; // 数据目录数组
 } IMAGE_OPTIONAL_HEADER;


其中比较重要的是这几个字段

ImageBase字段，表示可执行文件映像的基址，即程序加载到内存时的起始地址。

addressofentrypoint字段，程序入口点的地址。对于 DLL，通常是导出的函数地址，对于 EXE 是程序的启动入口。

另外重要的是sectionalignment和filealignment字段，它们分别表示节和文件的对齐大小。

？对齐？什么对齐？

现代的 CPU 在读取内存时，通常会一次读取一个最小的数据块，并将它存储在寄存器中。对于x86和x64架构的CPU，最小的数据块通常是字(word)、双字(dword)或四字(quadword)，也就是2、4、8字节。如果数据存储在内存中的起始地址没有按照合适的边界对齐，那么 CPU 可能需要多次访问内存，从而影响性能

由此，数据在内存中对齐的原则也就呼之欲出了，即

每种数据类型在内存中应存放在一个能够满足其对齐要求的地址上。对齐要求（Alignment Requirement）通常是该数据类型大小的倍数

同时，PE文件在执行的时候，映射到内存中的结构布局与该文件在磁盘中存储时的结构布局是一致的。Windows 装载器（又称 PE 装载器）在载入一个 PE 文件时，把磁盘中的 PE 文件映射到进程的内存地址空间，是一种从磁盘偏移地址到内存偏移地址的转换。

而问题在于

有的PE文件进程空间中的对齐大小和磁盘中的对齐大小有区别，进程空间当中的对齐大小默认值为系统页面大小 0x1000 B（4KB），在磁盘中存储时的对齐大小默认值为磁盘页面大小 0x200 B（512B）

所以我们可以知道filealignment决定了每个节的大小的位置都会按照 FileAlignment 进行对齐，其图中的值为512也就知道所有节在磁盘中的起始位置需要对齐到的边界为512的倍数。同样，sectionalignment决定了每个节在内存中的起始地址需要对齐到的边界，由图中其值为4096也就知道所有节在内存中的起始地址必须是 4096 的倍数

还有一个是DataDirectory字段，是一个数组，它的每个条目都是 IMAGE_DATA_DIRECTORY 类型的结构

    typedef struct _IMAGE_DATA_DIRECTORY {

  1. 节表

节表(Section Header)描述了程序中的各个节（Section），由 IMAGE_SECTION_HEADER 结构构成

下面为IMAGE_SECTION_HEADER结构的定义

    typedef struct _IMAGE_SECTION_HEADER {
            BYTE    Name[8];                 // 节的名称，最多 8 字符
            union {
                    DWORD   PhysicalAddress;    // 物理地址（对于磁盘文件）
                    DWORD   VirtualSize;        // 虚拟大小（对于内存映像）
            };
            DWORD   VirtualAddress;          // 节的虚拟地址（相对于程序的 ImageBase）
            DWORD   SizeOfRawData;           // 节数据的大小（从文件中提取的原始数据大小）
            DWORD   PointerToRawData;        // 从文件开始处到节原始数据的偏移量
            DWORD   PointerToRelocations;    // 相对地址，指向节的重定位数据
            DWORD   PointerToLinenumbers;    // 相对地址，指向节的源代码行号数据
            WORD    NumberOfRelocations;     // 该节的重定位项的数量
            WORD    NumberOfLinenumbers;     // 该节的源代码行号数据项的数量
            DWORD   Characteristics;         // 节的特性标志
            } IMAGE_SECTION_HEADER;

一些字段注释已经说明了，但有几个字段需要注意一下

  • Name字段，是一个8字节的数组，通常是ASCII字符串，最多只能有8个字符(如果节的名称超过8个字符，通常会被截断),用于储存节的名称{.text（代码段）、.data（数据段）、.rsrc（资源段）}
  • VirtualAddress字段，是一个相对虚拟地址（RVA），表示节在内存中的起始地址，相对于程序的 ImageBase。（这个地址是相对于程序加载到内存中的基地址（ImageBase）的偏移量。也就是说，这是节在内存中的位置，而不是文件中的位置。）
  • Characteristics字段，也是一个位域，内容如下图
